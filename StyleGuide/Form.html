vMOX Standard Form

Created June 5 2018

Composed by Craig Graham

Key Points - Forms:
  • vMOX form layout follows F-Centered Design to increase completion speed
  • The layout follows standard best practices, as documented by the UX Collective
  • Mockup can be viewed at: vMOX Standard Form Example Mockup

The vMOX Standard Form was designed to make filling out a form as painless on a user as possible. This is accomplished by following UX Form best practices such as making hint text visible at all times, aligning the inputs below the label, putting options into radio buttons whenever possible instead of in dropdowns, etc.

Standard Form Layout

Form Title/Header

Each form in vMOX should have a title to give the user some context around the form they will be filling out. Whenver it makes sense, it is helpful to include an icon to give further context. There are additional cases where it makes sense to include a call to action for the form at the title level, such as a "Preview" function.

Standard Form Title with Image
Input Section

vMOX forms are comprised of one or more input sections. A basic input section includes a title, always-visible hint text and a input field of some type for the user to complete. The input field will vary based on the requirements of the question with some example types being, free-form text fields, text-area fields, scale selections, etc. These input sections should follow the Law of Common Region by grouping the title, hint and input close together while providing adequate and equal spacing between each input section to make for a clear distinction.

Form Section Header

When a form starts to become long it sometimes make sense to break the form into different sections. The advantage is that it groups similar inputs together for the user and gives the user a sense that they are making progress while they work their way through filling out a lot of inputs. The section header is used to signal to the user that a new section of the form has started. The title should be very descriptive, but still short enough that it fits on one line.

Section Header
Continue Bar (Call to Action Fixed Position Bar)

Our forms all use a fixed/"Sticky" position footer to display the forms submit/continue call to action. This bar should always be visible to the user so they can know what they will need to do in order to complete the form. The Continue/Submit button should be disabled on the page load and should only be enabled when the form is completed and validated.

Continue Call to Action Fixed Bar
Form Validation

Whenever possible user input should be validated in real-time as the user types. A use should not have to click on submit/continue in order to see that they have entered a field incorrectly. For example, users should be notified if the value they have entered has been formatted incorrectly (phone numbers, email address, dates, numbers, etc) or is not available (username taken, company doesn't exist, etc). We should provide this feedback to the user by displaying a grey X that turns into a red X if there is an error condition or a green check mark if the condition is met.
